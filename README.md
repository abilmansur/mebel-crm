# Мебель CRM — MVP

CRM для мебельных цехов: единый инбокс заявок (WhatsApp/Telegram/звонки/сайт) + канбан-доска заказов
+ калькулятор сметы по материалам (ЛДСП/МДФ), фурнитуре и комментарию к заказу.
Каждый цех — отдельный аккаунт с изолированными данными через Supabase Auth + Row Level Security.

## Локальный запуск (без Supabase, на демо-данных)

```bash
npm install
npm run dev
```

Открой http://localhost:3000 — приложение работает на встроенных демо-данных
из `lib/demoData.ts`, без входа и без настройки базы. Удобно, чтобы сразу показать прототип клиенту.

## Подключение реальной базы и авторизации (Supabase)

1. Создай проект на [supabase.com](https://supabase.com) (бесплатный тариф достаточен для старта)
2. В SQL Editor выполни содержимое `supabase/schema.sql` — создаст таблицы и политики RLS
3. В Authentication → Providers убедись, что включён Email-провайдер
4. Для быстрого тестирования (без проверки почты): Authentication → Settings → выключи
   "Confirm email" — тогда после регистрации сразу будет сессия. Для продакшена лучше оставить включённым
5. Скопируй `.env.local.example` в `.env.local` и вставь `Project URL` и `anon public key` из
   Settings → API в Supabase
6. Перезапусти `npm run dev`

## Как это работает

- При первом заходе без сессии пользователя перекидывает на `/login`
- На `/login` можно зарегистрировать новый цех (email + пароль + название цеха) или войти в существующий
- При регистрации автоматически создаётся `workspace` в базе и засеваются стартовые материалы
  (ЛДСП 16мм, ЛДСП 25мм, МДФ) — см. `lib/workspace.ts`
- RLS в `supabase/schema.sql` гарантирует, что цех A никогда не увидит заказы цеха Б,
  даже если оба используют одно и то же Supabase-приложение

## Telegram-интеграция инбокса

Каждый цех подключает **своего собственного Telegram-бота своим токеном** — не общий бот платформы
с диплинками, а полноценный бренд-бот цеха. Владелец делает это сам прямо в интерфейсе, без твоего участия.

### Что нужно один раз настроить разработчику (тебе)

1. **Service Role key в Supabase**: Settings → API → Project API keys → скопируй **service_role**
   (не anon!). Это секретный ключ, обходит RLS, нужен серверным API-роутам.
2. Добавь в переменные окружения (локально `.env.local`, на Vercel — Settings → Environment Variables):
   ```
   SUPABASE_SERVICE_ROLE_KEY=service_role_key_из_supabase
   ```
3. На Vercel — обязательно **Redeploy** после сохранения.
4. Выполни миграцию `supabase/migrations/005_bot_per_workspace.sql` в SQL Editor (на новых установках
   уже включено в `supabase/schema.sql`).

### Что дальше делает каждый владелец цеха сам, без разработчика

Во вкладке «Входящие» есть блок «Подключить своего Telegram-бота» с инструкцией прямо в интерфейсе:

1. Пишет [@BotFather](https://t.me/BotFather) в Telegram → `/newbot` → придумывает имя и username
   (должен заканчиваться на `bot`, например `dubrava_mebel_bot`)
2. Копирует токен, который пришлёт BotFather
3. Вставляет токен в поле в CRM → нажимает «Подключить»

Дальше приложение само: проверяет токен через Telegram API, регистрирует вебхук на адрес
`https://твой-домен.vercel.app/api/telegram-webhook/<workspace_id>` и сохраняет токен в базе
(только для этого цеха, изолировано через RLS). Никаких переменных окружения на каждого клиента
добавлять не нужно — это и есть весь смысл модели «свой бот на цех».

### Как это работает дальше

- Клиент пишет боту цеха → сообщение появляется во «Входящие» именно этого цеха
- Мастер отвечает клиенту прямо из CRM — ответ уходит в Telegram через API-роут `send-reply`,
  используя токен именно этого бота
- Кнопка «Отключить» рядом с подключённым ботом убирает вебхук и токен из базы

## ИИ-ассистент и двусторонний диалог

Теперь во «Входящих» — не список одиночных заявок, а настоящие диалоги: клиент пишет, мастер видит
всю переписку и может ответить прямо из CRM (ответ уходит в Telegram). Плюс можно настроить ИИ,
который либо подсказывает мастеру вариант ответа, либо отвечает клиенту сам.

### Настройка ИИ-ассистента

1. Возьми API-ключ на [console.anthropic.com](https://console.anthropic.com)
2. Добавь `ANTHROPIC_API_KEY` в переменные окружения (Vercel + `.env.local`) → Redeploy
3. В приложении нажми на иконку 🤖 в шапке → впиши промпт: кто отвечает, как себя вести, что делать
   при вопросах о цене/сроках/материалах (как в примере из hh-bot: «Меня зовут Дина, я администратор...»)
4. Переключатель **«Отвечать клиентам автоматически»**:
   - **Выключен** (по умолчанию, безопаснее для старта) — ИИ только предлагает вариант ответа,
     мастер видит его в диалоге и решает, отправлять ли (кнопка «Использовать» подставляет текст в поле ответа)
   - **Включён** — ИИ отвечает клиенту сразу, без подтверждения. Использовать с осторожностью:
     если промпт неточный, ИИ может пообещать клиенту что-то не то

### Как отвечать клиенту вручную

Открой диалог во «Входящих» → внизу поле ввода → «Отправить». Ответ уходит клиенту в Telegram и
сохраняется в истории переписки. Кнопка «В заказ» в диалоге открывает создание заказа с уже
подставленным именем клиента, не удаляя историю чата.

### Нужна ли миграция базы

Если база создавалась до этого апдейта — выполни `supabase/migrations/004_ai_and_threads.sql`
в SQL Editor. На новых установках это уже включено в `supabase/schema.sql`.

## Что нового в этом апдейте

- **Вкладка «Каналы»** — подключение Telegram переехало из «Входящих» в отдельную вкладку.
  Instagram и WhatsApp показаны как заглушки "скоро" — технически не готовы, ждут Meta-верификации.
- **Готовый промпт по умолчанию** — при первом открытии настроек ИИ поле промпта уже не пустое,
  а содержит рабочий шаблон (`lib/defaultPrompt.ts`) с правилами общения, обработкой вопросов
  о цене/сроках и заглушками для персонализации под конкретный цех.
- **База знаний ИИ** — отдельная вкладка в настройках ассистента: список фактов (цены, сроки, FAQ)
  свободным текстом, добавляется к промпту при каждом ответе, чтобы ИИ не выдумывал цифры.
- **Примеры фото** — можно добавить фото (по ссылке) с ключевыми словами; если в сообщении клиента
  встретится ключевое слово, бот сам отправит фото в Telegram — независимо от режима автоответа.
- **Автообновление «Входящих»** — сообщения подтягиваются сами каждые 8 секунд, ручное обновление
  страницы больше не нужно. Открытый диалог тоже обновляется живьём.
- **Правильный счётчик** — бейдж на вкладке «Входящие» показывает количество ЧАТОВ с непрочитанными
  сообщениями, а не общее число сообщений. При открытии диалога сообщения помечаются прочитанными.

Если база создавалась раньше — выполни `supabase/migrations/006_channels_readstatus_knowledge.sql`.

## Что нужно доделать перед продажей нескольким цехам

1. **WhatsApp Business API / Instagram Direct.** Telegram уже подключён по-настоящему (см. раздел выше);
   остальные каналы пока нет — там нужен отдельный аккаунт в Meta Business и верификация, которая может
   занять от нескольких дней до нескольких недель (это не техническое, а бюрократическое ограничение).
2. **Транскрибация звонков** — отдельная подсистема (телефония + распознавание речи), не связана
   с текущим стеком, имеет смысл добавлять позже, когда будут живые клиенты.
3. **Оплата подписки.** Stripe/локальный эквайринг — на первых 3-5 клиентах можно выставлять счёт вручную.
4. **Уведомления о просрочке.** Поле `overdue` сейчас ставится вручную — нужен cron
   (Supabase Edge Function по расписанию), сравнивающий `due_date` с текущей датой.
5. **Восстановление пароля / смена email** — стандартные Supabase Auth флоу, не добавлены в MVP-интерфейс,
   но доступны через `supabase.auth.resetPasswordForEmail()` при необходимости.
6. **Перевод покрывает основной интерфейс** — часть плейсхолдеров в полях ввода намеренно оставлена
   на русском как разумный дефолт для первых клиентов; при желании дополняется в `lib/i18n.ts`.
7. **Удаление материала** пока не реализовано — только добавление и редактирование цены.
